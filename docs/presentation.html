Introduction

What is AI Code Maniac?

A self-hosted, multi-agent code analysis platform that orchestrates specialized AI agents in a coordinated pipeline to perform deep code analysis — instantly, consistently, and at scale.

Purpose

  • Automate repetitive code review tasks
  • Catch bugs before they reach production
  • Generate documentation from existing code
  • Assess commit history risk and readiness
  • Enable consistent quality checks at any scale

Problem Statement

The Code Review Gap

⏱

Manual Reviews Are Slow

Senior engineers spend hours reviewing code that automated tools could flag in seconds. High-value judgment is wasted on repetitive checks.

📉

Inconsistent Coverage

Code quality varies by reviewer. Security anti-patterns, performance issues, and design smells are caught inconsistently across teams.

🔇

No Documentation Trail

Legacy and undocumented codebases lack execution flow docs, requirement specs, and design reviews that are critical for onboarding.

AI Code Maniac solves this by orchestrating 27 specialized agents that run in parallel — delivering consistent, thorough analysis in under a minute.

Requirements

System Requirements

Functional Requirements

  • Accept code from GitHub, Gitea, and local files
  • Support single-file, multi-file (up to 50), and folder scan
  • Line-range targeting within any file
  • Run up to 27 analysis agents per job
  • 6-phase orchestrated pipeline with dependency awareness
  • Dual-phase security gate (Phase 0 regex + Phase 3 LLM)
  • Composite result caching by file hash + feature + language + prompt
  • 100+ built-in prompt templates across 13+ categories
  • Named settings profiles persisted across sessions
  • Custom prompt overrides (append or overwrite mode)
  • JSON + Markdown export for all result tabs
  • Commit-history risk and changelog analysis

Non-Functional Requirements

  • Non-blocking UI — background daemon thread + 2 s polling
  • Self-contained — no external database server (DuckDB)
  • 9-table DuckDB schema with versioned cache
  • Per-agent gating via ENABLED_AGENTS env var
  • Token-aware chunker for files exceeding LLM context limits
  • SQL injection prevention via parameterized queries
  • Credentials stored in .env, never logged or displayed
  • Thread-safe DuckDB singleton connection
  • Schema versioning to auto-invalidate stale cache entries
  • Graceful linter fallbacks when tools not installed
  • 156 automated tests covering agents, DB, tools, config

Design

Core Design Decisions

⚡
Parallel-first

Phase 1 agents use Python ThreadPoolExecutor — 13 Bedrock calls in parallel instead of sequential. Phase 2 starts only after Phase 1 completes to use its output as context.

🔒
Dual-Phase Security

Phase 0 regex pre-flight catches secrets before any LLM call. Phase 3 deep scan uses AI to find what regex misses. Two complementary layers of defense.

🗃
Cache-first

SHA-256 composite key on (file_hash + feature + language + prompt). Same file + same settings = instant cache hit. Schema versioning forces re-analysis on output schema changes.

🧩
Consistent Agent Contract

Every agent follows the same function signature. check_cache → miss → Bedrock → write_cache → add_history. Uniform pattern makes adding new agents trivial.

🔒
Isolated UI Thread

Streamlit never waits for analysis. A daemon thread runs the pipeline; the UI polls DuckDB every 2 seconds. The event loop is never blocked.

📦
Self-Contained

DuckDB needs no server. One file (data/arena.db) holds all 9 tables — jobs, cache, history, presets, profiles, chunks, events, metadata, snapshots.

Phase 1 · Static Analysis

Static Analysis Agent

Two-layer approach: a real linter (flake8/ESLint) plus AI semantic analysis.

Layer 1 — Linter
  • Python → flake8 subprocess, parses path:line:col: CODE message
  • JS/TS → ESLint subprocess with --format json
  • If not installed: skipped=True (graceful fallback)
Layer 2 — AI Semantic
  • Design smells and dead code detection
  • Security anti-patterns (hardcoded secrets, unsafe eval)
  • Performance issues (N+1 queries, unnecessary copies)
  • Maintainability issues and error-handling gaps
  • Feeds downstream Code Design + PR Comments agents
